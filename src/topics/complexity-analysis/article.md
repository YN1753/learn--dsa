# 复杂度分析 (Complexity Analysis)

## 概念解释

复杂度分析是计算机科学中最基础、最重要的概念之一。它提供了一套数学工具，让我们能够**不依赖具体硬件和编程语言**地衡量算法的效率。

想象你面前有两条路去学校：一条路需要走 1000 步，另一条路需要走 n × n 步（n 是你家到学校的公里数）。当距离很短时，两条路差不多；但当距离变大时，第二条路会变得极其漫长。复杂度分析就是帮你判断「当输入规模变大时，算法会变多慢」的方法。

### 渐近记号 (Asymptotic Notation)

我们使用三种主要的渐近记号来描述算法复杂度：

| 记号 | 名称 | 含义 | 直观理解 |
|------|------|------|----------|
| O(f(n)) | Big-O | 渐近上界 | 运行时间**不超过** f(n) 的量级 |
| Ω(f(n)) | Big-Ω | 渐近下界 | 运行时间**至少是** f(n) 的量级 |
| Θ(f(n)) | Big-Θ | 渐近紧确界 | 运行时间**恰好是** f(n) 的量级 |

**重要区分**：Big-O 描述的是数学上的界，不是「最坏情况」；Big-Ω 描述的也不是「最好情况」。一个算法的最坏情况可以用 Big-O 来表示，但 Big-O 本身只是上界。

### 常见复杂度等级

从快到慢排列：

| 复杂度 | 名称 | 例子 | n=1000 时的运算次数 |
|--------|------|------|---------------------|
| O(1) | 常数 | 数组按索引访问 | 1 |
| O(log n) | 对数 | 二分查找 | ≈ 10 |
| O(n) | 线性 | 遍历数组 | 1,000 |
| O(n log n) | 线性对数 | 归并排序 | ≈ 10,000 |
| O(n²) | 平方 | 冒泡排序 | 1,000,000 |
| O(2ⁿ) | 指数 | 暴力枚举所有子集 | 约 10³⁰¹ |
| O(n!) | 阶乘 | 暴力枚举所有排列 | 约 10²⁵⁶⁷ |

## 为什么重要

### 1. 选择正确的算法

面对同一个问题，往往有多种解法。复杂度分析帮你做出明智的选择：

- 在 100 万个元素中查找：线性查找 O(n) 最多需要 100 万次，二分查找 O(log n) 最多只需 20 次
- 对 10 万个元素排序：冒泡排序 O(n²) 可能需要 100 亿次操作，归并排序 O(n log n) 只需约 170 万次

### 2. 预测性能

复杂度告诉你当数据规模增大时，程序会变慢多少：

- 如果你的算法是 O(n)，数据量翻倍，运行时间也翻倍
- 如果你的算法是 O(n²)，数据量翻倍，运行时间变成 4 倍
- 如果你的算法是 O(log n)，数据量翻倍，运行时间只增加 1 次操作

### 3. 面试与竞赛

复杂度分析是技术面试和算法竞赛的必备知识。面试官会要求你分析代码的时间和空间复杂度，这是评估你算法能力的基本标准。

## 核心原理

### 如何分析时间复杂度

分析时间复杂度的基本原则：

1. **只看最高阶项**：O(n² + n) = O(n²)，因为当 n 足够大时，n² 远大于 n
2. **忽略常数系数**：O(3n) = O(n)，因为常数不影响增长趋势
3. **加法规则**：顺序执行的代码，取最大的复杂度
4. **乘法规则**：嵌套循环，复杂度相乘

### 分析循环

**单层循环**：循环次数 × 每次循环的复杂度

```typescript
// O(n) — 循环 n 次，每次 O(1)
for (let i = 0; i < n; i++) {
  console.log(i)
}
```

**嵌套循环**：外层 × 内层

```typescript
// O(n²) — 外层 n 次，内层 n 次
for (let i = 0; i < n; i++) {
  for (let j = 0; j < n; j++) {
    console.log(i, j)
  }
}
```

**减半循环**：O(log n)

```typescript
// O(log n) — 每次减半，共 log₂n 次
let i = n
while (i > 1) {
  i = Math.floor(i / 2)
}
```

**不等循环**：需要具体计算

```typescript
// O(n²) — 外层 n 次，内层 i 次，总次数 = 1+2+...+n = n(n+1)/2
for (let i = 0; i < n; i++) {
  for (let j = 0; j < i; j++) {
    console.log(i, j)
  }
}
```

### 主定理 (Master Theorem)

对于递归算法，主定理提供了快速分析的工具。如果递归关系为 T(n) = aT(n/b) + O(n^d)：

| 条件 | 结果 |
|------|------|
| d > log_b(a) | T(n) = O(n^d) |
| d = log_b(a) | T(n) = O(n^d · log n) |
| d < log_b(a) | T(n) = O(n^(log_b(a))) |

**例子**：归并排序 T(n) = 2T(n/2) + O(n)，a=2, b=2, d=1，因为 d = log_2(2) = 1，所以 T(n) = O(n log n)。

### 空间复杂度

空间复杂度衡量算法运行所需的**额外内存空间**（不包括输入数据本身）。

```typescript
// O(1) 空间 — 只用了固定几个变量
function sum(arr: number[]): number {
  let total = 0
  for (let i = 0; i < arr.length; i++) {
    total += arr[i]
  }
  return total
}

// O(n) 空间 — 创建了一个新数组
function double(arr: number[]): number[] {
  const result: number[] = []
  for (let i = 0; i < arr.length; i++) {
    result.push(arr[i] * 2)
  }
  return result
}
```

## 可视化说明

在可视化面板中，你可以直观地看到不同复杂度函数的增长曲线：

- **增长曲线图**：横轴为输入规模 n，纵轴为运算次数，展示 O(1)、O(log n)、O(n)、O(n log n)、O(n²) 的增长趋势
- **交互式滑块**：拖动滑块改变 n 的大小，实时观察各复杂度的变化
- **代码分析**：展示常见循环模式及其对应的复杂度
- **对比条形图**：在同一 n 值下，直观比较不同复杂度的实际运算次数

通过这些可视化，你会深刻理解为什么 O(n²) 在大数据量下不可接受，以及 O(log n) 为什么如此高效。

## 常见错误

### 1. 混淆 Big-O 与「最坏情况」

Big-O 是数学上的**上界**，不是「最坏情况」的代名词。一个算法的最好情况也可以用 Big-O 表示。正确的说法是：「最坏情况的时间复杂度是 O(n)」，而不是「Big-O 就是最坏情况」。

### 2. 忽略常数因子在实际中的影响

理论上 O(n) 和 O(2n) 都是 O(n)，但在实际运行中，2n 确实比 n 慢一倍。当 n 较小时，常数因子可能比复杂度等级更重要。例如，在小数组上，插入排序 O(n²) 可能比归并排序 O(n log n) 更快，因为插入排序的常数因子更小。

### 3. 循环分析中的 off-by-one 错误

```typescript
// 这是 O(n) 还是 O(n-1)？
for (let i = 1; i < n; i++) {
  // ...
}
// 答案：O(n)。n-1 和 n 在渐近意义下相同
```

### 4. 忘记考虑递归的空间复杂度

递归调用会占用调用栈空间。一个递归深度为 n 的算法，空间复杂度至少是 O(n)，即使没有显式创建新数组。

### 5. 误判嵌套循环的复杂度

```typescript
// 这不是 O(n²)！
for (let i = 0; i < n; i++) {
  for (let j = 0; j < 3; j++) {  // 内层固定 3 次
    console.log(i, j)
  }
}
// 实际是 O(3n) = O(n)，因为内层是常数
```

## 实际应用

### 1. 算法选择

在实际开发中，复杂度分析指导你选择合适的算法：

- 需要排序 100 万条数据？选 O(n log n) 的排序算法（如快速排序、归并排序），而不是 O(n²) 的冒泡排序
- 需要频繁查找？用哈希表 O(1) 或二分查找 O(log n)，而不是线性扫描 O(n)
- 需要处理图的最短路？稀疏图用 Dijkstra + 堆 O((V+E)log V)，稠密图用朴素 Dijkstra O(V²)

### 2. 性能瓶颈分析

当程序运行缓慢时，复杂度分析帮你定位问题：

- 如果你的算法是 O(n²)，而数据量达到了 10^5，那么大约需要 10^10 次操作——这就是慢的原因
- 优化方向：能否将复杂度降到 O(n log n) 或 O(n)？

### 3. 空间与时间的权衡

很多算法问题存在时间和空间的权衡：

- 哈希表用 O(n) 空间换取 O(1) 查找时间
- 动态规划用 O(n) 空间避免重复计算
- 原地算法用更多时间换取 O(1) 空间

## 总结

复杂度分析是算法学习的基石：

- **核心工具**：Big-O（上界）、Big-Ω（下界）、Big-Θ（紧确界）
- **分析方法**：只看最高阶、忽略常数、顺序取最大、嵌套相乘
- **常见等级**：O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!)
- **递归分析**：使用主定理快速判断
- **空间复杂度**：关注额外内存使用，注意递归栈

掌握复杂度分析，你就拥有了评判算法优劣的标尺，这是成为优秀程序员的第一步。在后续学习每一种数据结构和算法时，我们都会分析其时间复杂度和空间复杂度——这将贯穿整个算法学习之旅。
