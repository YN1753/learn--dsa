# 红黑树 (Red-Black Tree)

## 概念解释

红黑树是一种**自平衡二叉搜索树**，由 Rudolf Bayer 在 1972 年发明，当时称为"对称二叉 B 树"。后来 Leo Guibas 和 Robert Sedgewick 在 1978 年将其改进并命名为"红黑树"。红黑树通过为每个节点添加**颜色标记**（红色或黑色），并遵循五条严格的性质来维持树的近似平衡。

### 红黑树的五条性质

红黑树必须始终满足以下五条性质，任何操作都不能违反这些性质：

**性质 1：每个节点是红色或黑色**

每个节点都有一个颜色属性，要么是红色，要么是黑色。这是红黑树最基本的属性。

**性质 2：根节点是黑色**

红黑树的根节点必须是黑色。这条性质确保了从根开始的路径分析有一个统一的起点。

**性质 3：所有叶子（NIL）节点是黑色**

红黑树中的叶子节点指的是空指针（NIL），它们被视为黑色节点。在实际实现中，通常使用一个哨兵节点（sentinel）来表示所有 NIL 叶子。

**性质 4：红色节点的子节点必须是黑色**

如果一个节点是红色的，那么它的两个子节点都必须是黑色。换句话说，**不能有两个连续的红色节点**。这条性质限制了红黑树中最长路径的长度。

**性质 5：从任意节点到其所有后代叶子的路径上，黑色节点数目相同**

对于树中的每个节点，从该节点到其所有后代叶子（NIL）的路径上，包含的黑色节点数量必须相同。这个数量称为该节点的**黑高**（Black Height）。

### 红黑树示例

```
        [10]B
       /    \
     [5]R   [15]B
     / \      \
   [3]B [7]B  [20]R
```

上面这棵树满足所有五条性质：
- 性质 1：每个节点都有颜色标记
- 性质 2：根节点 10 是黑色
- 性质 3：所有 NIL 叶子是黑色
- 性质 4：红色节点 5 的子节点 3 和 7 都是黑色；红色节点 20 的子节点都是黑色 NIL
- 性质 5：从根到任何 NIL 的路径上都有 2 个黑色节点

### 为什么这些性质能保证平衡？

性质 4 和性质 5 共同作用，保证了红黑树的高度上界。由于不能有连续的红色节点，最长路径（交替红黑）不超过最短路径（全黑）的 2 倍。数学上可以证明，n 个节点的红黑树高度不超过 **2 * log₂(n+1)**。

## 为什么重要

### 1. 标准库的核心实现

红黑树是工业界最广泛使用的平衡二叉搜索树。几乎所有主流编程语言的标准库都使用红黑树来实现有序容器：

- **Java**：`TreeMap`、`TreeSet`、`ConcurrentSkipListMap` 的底层实现
- **C++ STL**：`std::map`、`std::multimap`、`std::set`、`std::multiset` 的底层实现
- **C#**：`SortedDictionary`、`SortedSet` 的底层实现

选择红黑树而非 AVL 树的原因在于：红黑树的插入和删除操作旋转次数有常数上界（插入最多 2 次旋转，删除最多 3 次旋转），而 AVL 树的删除可能需要 O(log n) 次旋转。

### 2. 操作系统内核

Linux 内核广泛使用红黑树：

- **CFS 调度器**（Completely Fair Scheduler）：使用红黑树管理进程的虚拟运行时间（vruntime），保证调度的公平性。红黑树的 O(log n) 插入和删除确保了调度器的高效运作。
- **内存管理**：Linux 的虚拟内存区域（VMA）管理使用红黑树，支持快速的地址区间查找和合并。
- **IO 调度器**：某些 IO 调度算法使用红黑树管理请求队列。

### 3. 高性能服务器

- **Nginx**：使用红黑树管理定时器事件，支持高效的超时检测和处理。
- **Epoll**：Linux 的 epoll 在某些实现中使用红黑树管理文件描述符。

### 4. 数据库系统

许多数据库系统的内存索引结构使用红黑树或其变种，用于维护有序数据和支持范围查询。

## 核心原理

### 插入操作

红黑树的插入操作分为两个阶段：

**阶段一：按 BST 规则插入**

与普通二叉搜索树一样，从根节点开始比较，找到合适的空位插入新节点。新插入的节点**总是染成红色**。为什么是红色？因为插入红色节点不会违反性质 5（黑高不变），最多只会违反性质 4（可能产生连续红色节点），修复起来更简单。

**阶段二：修复红黑树性质**

插入红色节点后，可能违反性质 4（父节点也是红色，产生"双红"冲突）。修复策略根据叔叔节点的颜色分为三种情况：

**情况 1：叔叔节点是红色**

```
      [G]B              [G]R          或        [P]R
      / \      →        / \          →         /    \
   [P]R  [U]R       [P]B  [U]B             [G]B     [新]R
    |                                   (其他调整)
  [新]R
```

操作：将父节点和叔叔节点染黑，祖父节点染红，然后以祖父节点为起点继续向上检查。这是**颜色翻转**（color flip），不需要旋转。

**情况 2：叔叔节点是黑色（或 NIL），且新节点是父节点的右子节点（LR 型）**

操作：对父节点执行**左旋**，转化为情况 3。

**情况 3：叔叔节点是黑色（或 NIL），且新节点是父节点的左子节点（LL 型）**

```
      [G]B              [P]B
      / \      →       /    \
   [P]R  [U]B       [新]R   [G]R
    /                          \
 [新]R                        [U]B
```

操作：将父节点染黑，祖父节点染红，然后对祖父节点执行**右旋**。

插入操作最多需要 **2 次旋转** 即可恢复所有性质。

### 删除操作

红黑树的删除比插入复杂得多。删除过程：

1. **按 BST 规则删除**：找到要删除的节点，如果它有两个子节点，用中序后继替换
2. **判断是否需要修复**：如果被删除的节点或替换节点是红色，删除后不违反性质；如果是黑色，则会减少某条路径上的黑高，需要修复

删除修复有多种情况，根据兄弟节点及其子节点的颜色组合来决定操作：

- **兄弟是红色**：通过旋转和颜色调整，转化为兄弟是黑色的情况
- **兄弟是黑色，且兄弟的两个子节点都是黑色**：将兄弟染红，向上继续检查
- **兄弟是黑色，兄弟的远侄子是红色**：旋转并调整颜色，修复完成
- **兄弟是黑色，兄弟的近侄子是红色，远侄子是黑色**：旋转兄弟，转化为上一种情况

删除操作最多需要 **3 次旋转**，但可能需要 O(log n) 次颜色调整。

### 与 AVL 树的对比

| 特性 | AVL 树 | 红黑树 |
|------|--------|--------|
| 平衡程度 | 严格平衡（高度差 <= 1） | 近似平衡（最长路径 <= 2 * 最短路径） |
| 树高上界 | 1.44 * log₂n | 2 * log₂n |
| 查找效率 | 更快（树更矮） | 略慢 |
| 插入旋转 | 最多 2 次 | 最多 2 次 |
| 删除旋转 | 最多 O(log n) 次 | 最多 3 次 |
| 实现复杂度 | 中等 | 较高 |
| 适用场景 | 查找密集 | 插入/删除频繁 |

### 时间复杂度

| 操作 | 平均 | 最坏 |
|------|------|------|
| 查找 | O(log n) | O(log n) |
| 插入 | O(log n) | O(log n) |
| 删除 | O(log n) | O(log n) |
| 空间 | O(n) | O(n) |

## 可视化说明

红黑树的可视化重点展示以下内容：

- **树结构**：节点之间的父子关系，红色节点用红色圆圈表示，黑色节点用黑色/深色圆圈表示
- **颜色信息**：每个节点旁边标注 R（红）或 B（黑）
- **NIL 叶子**：用灰色小方块表示 NIL 哨兵节点
- **插入过程**：逐步展示新节点插入、颜色翻转、旋转的完整过程
- **双红冲突**：用闪烁或高亮标记违反性质 4 的连续红色节点
- **修复过程**：用动画展示颜色变化和旋转操作

通过可视化，可以直观地理解：
- 为什么新节点总是红色（减少冲突概率）
- 颜色翻转如何在不旋转的情况下修复性质
- 旋转如何同时调整结构和颜色

## 常见错误

### 1. 插入后忘记修复红黑树性质

最常见的错误是按照 BST 规则插入节点后，没有从插入点向上回溯检查并修复性质 4（双红冲突）。即使新节点是红色，如果其父节点也是红色，就必须进行修复。

### 2. 旋转时忘记更新颜色

红黑树的旋转不仅改变树的结构，还需要正确调整节点的颜色。例如，在 LL 型修复中，旋转后必须将父节点染黑、祖父节点染红。如果只做旋转不改颜色，会违反红黑树性质。

### 3. 删除时混淆"双重黑色"的处理

删除黑色节点可能导致某条路径黑高减少 1，产生所谓的"双重黑色"问题。修复过程需要根据兄弟节点及其子节点的颜色组合来选择不同策略，这是红黑树最复杂的部分。

### 4. 忘记处理 NIL 哨兵节点

性质 3 要求所有叶子（NIL）是黑色。在实现中，通常使用一个共享的黑色哨兵节点来代替所有 NIL 指针，而不是使用 null。忘记正确处理哨兵节点会导致性质检查失败。

### 5. 根节点颜色未维护

性质 2 要求根节点是黑色。在插入修复过程中，可能将根节点的父节点染红并上移。修复结束后必须确保根节点被强制设为黑色。

## 实际应用

### 1. Java TreeMap / TreeSet

Java 的 `TreeMap` 和 `TreeSet` 底层使用红黑树实现。`TreeMap` 提供按键有序的映射，支持 `firstKey()`、`lastKey()`、`headMap()`、`tailMap()` 等有序操作。选择红黑树而非 AVL 树的原因是 TreeMap 需要高效的插入和删除操作，而红黑树的旋转次数更可控。

```java
TreeMap<Integer, String> map = new TreeMap<>();
map.put(3, "三");
map.put(1, "一");
map.put(4, "四");
// 底层红黑树保证按键有序：1, 3, 4
```

### 2. C++ std::map / std::set

C++ STL 的 `std::map`、`std::multimap`、`std::set`、`std::multiset` 均使用红黑树实现。这些容器提供 O(log n) 的查找、插入、删除，并保持元素有序。

```cpp
std::map<int, std::string> m;
m[3] = "three";
m[1] = "one";
// 底层红黑树保证迭代器按 key 有序访问
```

### 3. Linux 内核 CFS 调度器

Linux 的 CFS（Completely Fair Scheduler）使用红黑树管理所有可运行进程。每个进程的虚拟运行时间（vruntime）作为红黑树的键，vruntime 最小的进程在树的最左侧。调度器总是选择最左侧的节点运行，保证了公平性。

### 4. Nginx 定时器管理

Nginx 使用红黑树管理所有定时器事件（如连接超时、会话过期）。定时器的过期时间作为键，红黑树保证了定时器的有序管理和 O(log n) 的添加/删除/查找操作。

### 5. Epoll 事件管理

Linux 的 epoll 在某些实现中使用红黑树管理文件描述符集合，支持高效的事件注册、删除和查找操作。

## 总结

红黑树是工业界最重要的自平衡二叉搜索树，通过**颜色标记**和**五条性质**维持近似平衡。它的核心优势在于：

- **旋转次数有常数上界**：插入最多 2 次旋转，删除最多 3 次旋转，远优于 AVL 树
- **保证 O(log n) 的所有操作**：树高不超过 2 * log₂(n+1)
- **广泛的实际应用**：Java TreeMap、C++ std::map、Linux CFS、Nginx 定时器

红黑树的插入修复核心是**颜色翻转**和**旋转**的组合，删除修复则更加复杂，需要处理多种兄弟节点颜色组合的情况。理解红黑树需要掌握五条性质的含义、插入修复的三种情况、以及删除修复的四种情况。

与 AVL 树的权衡：**查找密集选 AVL 树，插入/删除频繁选红黑树**。在实际工程中，红黑树因其更稳定的修改性能而成为标准库的首选实现。
