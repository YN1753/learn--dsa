# 跳表 (Skip List)

## 概念解释

跳表是一种**概率性数据结构**，它通过在有序链表的基础上建立多层索引来加速查找操作。可以将跳表理解为一个拥有"快速通道"的有序链表——底层是包含所有元素的完整有序链表，而上层则是越来越稀疏的索引层，类似于书本的目录结构。

### 多层链表结构

跳表的核心思想是**空间换时间**。在一个普通的有序链表中，查找某个元素需要从头到尾逐个遍历，时间复杂度为 O(n)。跳表通过在原始链表之上建立多级"快速通道"（索引层），使得查找操作可以跳过大量不必要的节点。

- **第 0 层（底层）**：包含所有元素的完整有序链表
- **第 1 层**：从底层每隔一个元素抽取一个作为索引
- **第 2 层**：从第 1 层每隔一个元素抽取一个作为索引
- **更高层**：依此类推，层越高元素越稀疏

### 概率性平衡

与平衡二叉搜索树（如 AVL 树、红黑树）不同，跳表**不依赖严格的平衡条件**。它的平衡是通过随机化实现的：当插入一个新元素时，通过抛硬币（或随机数生成器）决定该元素能提升到多少层索引。

这种概率性方法带来了显著的实现优势：
- 无需复杂的旋转操作来维护平衡
- 代码实现简单直观
- 期望时间复杂度为 O(log n)

### 快速通道（Express Lanes）

跳表的高层索引可以看作是"快速通道"。搜索时从最高层开始，沿着当前层向右移动，当遇到大于目标值的节点时向下移动到下一层继续搜索。这种"先走高速再走小路"的策略大大减少了需要访问的节点数量。

## 为什么重要

### Redis 中的有序集合

跳表最著名的应用是 **Redis 的有序集合（Sorted Set）**。Redis 选择跳表而非红黑树来实现 ZSET，原因包括：

1. **实现简单**：跳表的代码量远少于红黑树，更易于维护和调试
2. **范围查询高效**：跳表天然支持高效的范围查询操作（ZRANGEBYSCORE）
3. **并发友好**：跳表的细粒度锁比红黑树更容易实现
4. **内存局部性**：跳表的顺序访问模式对 CPU 缓存更友好

### 比平衡树更简单

实现一个红黑树需要处理大量的旋转和颜色翻转逻辑，而跳表的核心操作（插入、删除、查找）只需几十行代码。在工程实践中，简单性意味着更少的 Bug 和更低的维护成本。

### 期望 O(log n) 时间复杂度

跳表的查找、插入和删除操作的**期望时间复杂度**均为 O(log n)。虽然最坏情况下退化为 O(n)，但这种情况发生的概率极低（与抛硬币连续出现 n 次正面的概率相当）。

## 核心原理

### 底层：有序链表

跳表的最底层是一个普通的有序链表，每个节点存储一个值和指向下一个节点的指针。这是数据的"真相层"，所有元素都存在于这一层。

### 高层：稀疏索引

每一层都是下一层的一个子集。第 k 层大约包含第 k-1 层一半的元素。对于包含 n 个元素的跳表：
- 第 0 层有 n 个节点
- 第 1 层约有 n/2 个节点
- 第 2 层约有 n/4 个节点
- 第 k 层约有 n/2^k 个节点

总层数约为 log₂(n)。

### 搜索过程

搜索从最高层的头节点开始：

```
1. 在当前层向右移动，直到下一个节点的值大于等于目标值
2. 如果下一个节点的值等于目标值，搜索成功
3. 如果下一个节点的值大于目标值，向下移动到下一层
4. 重复步骤 1-3，直到找到目标或到达底层且无法继续
```

### 插入与随机提升

插入操作分为两步：

1. **查找插入位置**：按照搜索过程找到底层的插入位置
2. **随机提升**：新节点在底层插入后，通过抛硬币决定是否提升到上一层。每次抛硬币有 50% 的概率提升，直到抛到反面停止

```
抛硬币结果：正 正 正 反
新节点出现在：第 0 层、第 1 层、第 2 层、第 3 层（共 4 层）
```

## 常见错误

### 不理解概率性本质

初学者常误以为跳表的性能不稳定。实际上，跳表的期望性能非常稳定：
- 期望层数：O(log n)
- 期望每层访问节点数：O(1)
- 总期望时间复杂度：O(log n)

只有当随机数生成器质量极差时，性能才会退化。

### 与确定性结构直接比较

跳表是概率性结构，红黑树是确定性结构。直接比较"谁更快"没有意义，因为：
- 跳表的期望 O(log n) 在实践中与红黑树的最坏 O(log n) 表现相当
- 跳表的优势在于实现简单性和并发友好性
- 红黑树的优势在于严格的最坏情况保证

### 层级生成偏差

实现跳表时最常见的错误是**层级生成不均匀**。正确的做法是：
- 每次以概率 p（通常为 0.5）决定是否继续提升
- 不要预先确定层数，而是在每次插入时动态决定
- 设置最大层数限制（通常为 log₂(n) 的常数倍）

## 实际应用

### Redis ZSET

Redis 使用跳表实现有序集合的核心操作：
- `ZADD`：插入元素，O(log n)
- `ZSCORE`：查询分数，O(1)（配合哈希表）
- `ZRANK`：查询排名，O(log n)
- `ZRANGE`：范围查询，O(log n + k)

Redis 的实现中，每个跳表节点同时存在于一个哈希表中，实现 O(1) 的精确查找。

### 并发数据结构

跳表天然适合并发场景：
- 可以在不同层使用细粒度锁
- 无锁跳表（Lock-Free Skip List）使用 CAS 操作实现
- Java 的 `ConcurrentSkipListMap` 就是基于跳表的并发有序映射

### 内存数据库

许多内存数据库（如 LevelDB、RocksDB）在内存表（MemTable）中使用跳表：
- 插入和查找都是 O(log n)
- 支持高效的范围扫描
- 实现简单，便于序列化

## 总结

跳表是一种优雅的概率性数据结构，它通过随机化策略在有序链表之上建立多层索引，实现了 O(log n) 的期望查找性能。虽然它不提供最坏情况的严格保证，但其简单的实现、高效的范围查询能力和并发友好性，使其成为工程实践中的热门选择。Redis、LevelDB 等知名系统都采用跳表作为核心数据结构，证明了"简单即美"的设计哲学。

| 操作 | 期望时间复杂度 | 最坏时间复杂度 |
|------|---------------|---------------|
| 查找 | O(log n) | O(n) |
| 插入 | O(log n) | O(n) |
| 删除 | O(log n) | O(n) |
| 空间 | O(n) | O(n log n) |
