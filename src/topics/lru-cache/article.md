# LRU 缓存 (LRU Cache)

## 概念解释

LRU 是 **Least Recently Used**（最近最少使用）的缩写，是一种经典的缓存淘汰策略。当缓存空间有限时，LRU 策略会优先淘汰「最久未被访问」的数据，保留最近被使用的数据。

### 核心思想

LRU 基于一个简单而有效的假设：**如果数据最近被访问过，那么它将来被访问的概率也更高**。这被称为「时间局部性」（Temporal Locality），是计算机系统中普遍存在的访问模式。

想象你的书桌上只能放 5 本书。每当你要读一本新书时，如果书桌已满，你会把「最久没翻过的那本书」放回书架——这就是 LRU 策略的直觉。

### 数据结构组合

实现 LRU 缓存的关键在于同时获得两种能力：

- **O(1) 查找**：通过哈希表（HashMap），用 key 直接定位到对应的链表节点
- **O(1) 重排**：通过双向链表（Doubly Linked List），快速将访问的节点移到头部、快速删除尾部节点

这两种数据结构的组合是 LRU 缓存的精髓所在。

## 为什么重要

### 操作系统页面置换

在虚拟内存管理中，当物理内存不足时，操作系统需要选择一个页面换出到磁盘。LRU 是最常用的页面置换算法之一，它能有效减少页面缺失（Page Fault）的次数，提高系统性能。

### CPU 缓存

现代 CPU 的多级缓存（L1/L2/L3 Cache）在容量有限的情况下，需要高效的替换策略。虽然硬件层面通常使用近似 LRU 的策略（如伪 LRU），但其核心思想与 LRU 一致。

### Redis 与数据库

Redis 的内存淘汰策略中，`allkeys-lru` 和 `volatile-lru` 选项直接使用 LRU 算法。数据库系统（如 MySQL、PostgreSQL）的缓冲池管理也广泛采用 LRU 或其变体来决定哪些数据页应该留在内存中。

### CDN 与 Web 缓存

内容分发网络（CDN）和浏览器缓存使用 LRU 策略管理有限的存储空间，确保热门内容被保留在缓存中，冷门内容被淘汰。

### 面试经典题

LRU 缓存是技术面试中出现频率最高的设计题之一。它考察候选人对哈希表和链表的理解、O(1) 时间复杂度的设计能力，以及将多种数据结构组合解决实际问题的能力。

## 核心原理

### 数据结构设计

```
哈希表 (HashMap):  key -> 链表节点引用
双向链表 (Doubly Linked List):  HEAD <-> [最近使用] <-> ... <-> [最久未使用] <-> TAIL
```

- 链表头部（HEAD 之后）：最近使用的元素
- 链表尾部（TAIL 之前）：最久未使用的元素
- 哈希表：存储 key 到链表节点的映射

### get(key) 操作

```
1. 在哈希表中查找 key
2. 如果不存在，返回 -1
3. 如果存在：
   a. 获取对应的链表节点
   b. 将该节点从当前位置删除
   c. 将该节点插入到链表头部（标记为最近使用）
   d. 返回节点的 value
```

时间复杂度：O(1)（哈希查找 + 链表移动均为 O(1)）

### put(key, value) 操作

```
1. 如果 key 已存在：
   a. 更新节点的 value
   b. 将该节点移到链表头部
2. 如果 key 不存在：
   a. 创建新节点
   b. 在哈希表中添加 key -> 新节点的映射
   c. 将新节点插入链表头部
   d. 如果缓存已满（超过 capacity）：
      - 获取链表尾部节点（最久未使用）
      - 从哈希表中删除该节点的 key
      - 从链表中删除该节点
```

时间复杂度：O(1)（所有子操作均为 O(1)）

### 关键设计细节

1. **哨兵节点**：使用 HEAD 和 TAIL 两个哨兵节点可以避免处理空链表和边界情况的特殊逻辑
2. **双向链表**：必须是双向的，因为删除节点时需要访问前驱节点，单向链表无法 O(1) 完成
3. **哈希表存储节点引用**：不是存储 value，而是存储链表节点的引用（指针），这样才能 O(1) 定位并移动节点

## 常见错误

### get 操作忘记更新链表

最常见的实现错误是：get 操作只从哈希表中读取 value，而忘记将访问的节点移到链表头部。这会导致缓存的淘汰顺序与实际访问模式不一致，破坏 LRU 的正确性。

```
// 错误实现
get(key) {
  return hashmap.get(key).value  // 忘记移动到头部！
}

// 正确实现
get(key) {
  node = hashmap.get(key)
  moveToHead(node)  // 关键步骤
  return node.value
}
```

### 淘汰逻辑错误

另一个常见错误是淘汰链表头部节点（最近使用的）而非尾部节点（最久未使用的）。记住：头部 = 最近使用，尾部 = 最久未使用，淘汰时应删除尾部。

### 线程安全问题

在多线程环境中，LRU 缓存的 get 和 put 操作需要加锁保护。常见的问题包括：
- 读写竞争导致数据不一致
- 锁粒度过大导致性能下降
- 死锁（如在持有锁的情况下调用外部方法）

实际生产环境中通常使用读写锁（ReadWriteLock）或无锁并发方案。

### 哨兵节点处理不当

不使用哨兵节点时，需要大量边界条件判断（链表为空、只有一个节点等）。使用哨兵节点可以简化代码，但需要注意：
- 初始化时 HEAD.next = TAIL，TAIL.prev = HEAD
- 添加节点时在 HEAD 之后插入
- 删除节点时连接其前驱和后继

### 容量为 0 或 1 的边界情况

实现时需要考虑容量为 0（所有操作都无效）和容量为 1（每次插入都触发淘汰）的边界情况，确保代码的健壮性。

## 实际应用

### 操作系统页面置换

当物理内存不足时，操作系统需要将某些页面换出到磁盘。LRU 策略会选择最久未被访问的页面进行换出。由于精确 LRU 的实现成本较高（需要记录每次内存访问），实际系统通常使用近似 LRU 算法，如时钟算法（Clock Algorithm）。

### 数据库缓冲池

MySQL 的 InnoDB 存储引擎使用改进的 LRU 算法管理缓冲池（Buffer Pool）。它将 LRU 链表分为 young 区和 old 区，新读入的页面先放在 old 区头部，只有在一定时间后再次被访问时才移到 young 区，避免全表扫描等操作污染热点数据。

### Web 浏览器缓存

浏览器使用 LRU 策略管理 HTTP 缓存、DNS 缓存等。当缓存空间不足时，最近最少访问的资源会被淘汰。这确保了用户常访问的网站资源能快速加载。

### CDN 内容分发

CDN 节点的存储空间有限，使用 LRU（或其变体）策略决定哪些内容应该缓存在边缘节点上。热门内容（最近频繁访问）保留在缓存中，冷门内容被淘汰，以最大化缓存命中率。

### API 限流与会话管理

在分布式系统中，LRU 缓存常用于存储 API 限流的计数器和用户会话信息。通过设置合理的容量和过期策略，可以在内存消耗和性能之间取得平衡。

## 总结

LRU 缓存是数据结构设计的经典案例，它展示了如何巧妙地组合哈希表和双向链表两种基础数据结构，在 O(1) 时间复杂度内完成查找、插入和淘汰操作。

其核心设计思想——**用哈希表换取 O(1) 查找，用双向链表维护访问顺序**——在系统设计中具有广泛的适用性。从操作系统到数据库，从浏览器到 CDN，LRU 策略无处不在。

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| get | O(1) | 哈希查找 + 链表移动 |
| put | O(1) | 哈希插入/更新 + 链表操作 + 可能的淘汰 |
| 空间 | O(capacity) | 哈希表 + 双向链表各占 O(capacity) |

掌握 LRU 缓存的实现原理，不仅是面试的必备知识，更是理解计算机系统中缓存机制的重要基础。
