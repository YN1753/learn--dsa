# B树 (B-Tree)

## 概念解释

B树（B-Tree）是一种**自平衡的多路搜索树**，由 Rudolf Bayer 和 Edward McCreight 于 1970 年在波音研究实验室（Boeing Research Labs）提出。它的名字中的 "B" 可能代表 "Bayer"、"Boeing" 或 "Balanced"，至今没有官方定论。

B树是一种 **m 阶搜索树**（m-way search tree），具有以下性质：

- 每个节点最多有 **m 个子节点**
- 每个节点最多有 **m-1 个关键字**（键值）
- 每个非根节点至少有 **⌈m/2⌉ 个子节点**（即至少有 ⌈m/2⌉-1 个关键字）
- 根节点至少有 **2 个子节点**（除非它是叶子节点）
- **所有叶子节点在同一层**（绝对平衡）
- 节点内的关键字**有序排列**

例如，一棵 3 阶 B树（也叫 2-3 树）的每个节点可以有 2~3 个子节点和 1~2 个关键字：

```
        [10, 20]
       /    |    \
   [3,7] [15,18] [25,30]
```

每个节点包含一个有序的关键字数组和对应的子节点指针数组。对于节点中的关键字 k1 < k2 < ... < kn，子节点指针 p0, p1, ..., pn 满足：p0 指向的关键字都小于 k1，p1 指向的关键字都在 k1 和 k2 之间，以此类推。

## 为什么重要

B树之所以成为数据库和文件系统的核心数据结构，根本原因在于**磁盘 I/O 是最大的性能瓶颈**。

### 1. 最小化磁盘 I/O

磁盘的访问速度比内存慢 10 万倍以上。一次磁盘寻道（seek）大约需要 10ms，而一次内存访问只需 100ns。B树的设计目标就是**减少磁盘访问次数**：

- B树的每个节点通常对应一个**磁盘页**（如 4KB 或 16KB）
- 每个节点可以存储大量关键字（而非二叉树的 1 个）
- 树的高度极低，查找时只需访问少量节点

假设每个节点能存储 100 个关键字：
- 2 层 B树可存储约 1 万个记录
- 3 层 B树可存储约 100 万个记录
- 4 层 B树可存储约 1 亿条记录

也就是说，**在 1 亿条记录中查找任意一条，只需 4 次磁盘 I/O**！

### 2. 数据库索引

MySQL InnoDB 存储引擎使用 **B+树**（B树的变体）作为索引结构。每张表的主键索引是一棵 B+树，叶子节点存储完整的行数据。二级索引的叶子节点存储主键值。

InnoDB 默认的页大小是 16KB，假设主键为 8 字节的 bigint，指针为 6 字节，那么每个节点大约能存储 16384 / 14 ≈ 1170 个关键字。3 层 B+树就能索引约 1170 × 1170 × 16 ≈ 2000 万条记录。

### 3. 文件系统

许多文件系统使用 B树或其变体来管理目录和文件块：
- **NTFS**（Windows）使用 B+树管理文件索引
- **ext4**（Linux）使用 B树变体（HTree）管理目录
- **HFS+**（macOS）使用 B树管理文件系统元数据
- **ReiserFS** 使用 B+树存储所有数据

## 核心原理

### 查找操作

B树的查找类似于二叉搜索树的查找，但在每个节点内需要在多个关键字中进行搜索。

**时间复杂度：O(log_m n)**，其中 m 是阶数，n 是关键字总数。

```typescript
function search(node: BTreeNode, key: number): boolean {
  let i = 0
  // 在当前节点中找到第一个 >= key 的位置
  while (i < node.keys.length && key > node.keys[i]) {
    i++
  }
  // 找到了
  if (i < node.keys.length && key === node.keys[i]) {
    return true
  }
  // 如果是叶子节点，没找到
  if (node.isLeaf) {
    return false
  }
  // 递归在子节点中查找
  return search(node.children[i], key)
}
```

### 插入操作

插入操作是 B树最核心的操作，涉及**节点分裂**：

1. 从根节点开始，找到应该插入的叶子节点
2. 如果叶子节点未满（关键字数 < m-1），直接插入
3. 如果叶子节点已满，**分裂**该节点：
   - 将 m 个关键字从中间分开
   - 中间关键字上提到父节点
   - 左侧关键字留在原节点
   - 右侧关键字放入新节点
4. 如果父节点也满了，递归分裂
5. 最坏情况下，根节点分裂，树增高一层

```typescript
function insert(tree: BTree, key: number): void {
  const root = tree.root
  if (root.keys.length === MAX_KEYS) {
    // 根节点满了，需要分裂
    const newRoot = new BTreeNode(false)
    newRoot.children[0] = root
    splitChild(newRoot, 0)
    tree.root = newRoot
  }
  insertNonFull(tree.root, key)
}
```

### 删除操作

删除操作比插入更复杂，需要处理多种情况：

1. **在叶子节点中删除**：直接删除，如果关键字数低于下限，需要调整
2. **在内部节点中删除**：
   - 如果左兄弟有多余关键字，从左兄弟借一个（旋转）
   - 如果右兄弟有多余关键字，从右兄弟借一个（旋转）
   - 如果兄弟都没有多余关键字，**合并**两个节点
3. **递归调整**：合并后可能导致父节点关键字不足，需要递归调整

### 阶数 m 的含义

- **m 阶 B树**：每个节点最多有 m 个子节点
- 每个节点最多有 **m-1** 个关键字
- 每个非根节点至少有 **⌈m/2⌉** 个子节点（至少 ⌈m/2⌉-1 个关键字）
- 根节点至少有 **2** 个子节点（除非是叶子）

| 阶数 m | 每节点最大关键字 | 每节点最小关键字（非根） | 典型应用 |
|--------|----------------|----------------------|----------|
| 3 | 2 | 1 | 2-3 树，教学 |
| 4 | 3 | 2 | 2-3-4 树 |
| 100 | 99 | 49 | 小型数据库索引 |
| 1000+ | 999+ | 499+ | 大型数据库索引 |

## B+树 vs B树区别

B+树是 B树最重要的变体，也是实际数据库系统最常用的数据结构：

| 特性 | B树 | B+树 |
|------|-----|------|
| 数据存储位置 | 所有节点都存数据 | 只有叶子节点存数据 |
| 内部节点 | 存储关键字和数据 | 只存储关键字（纯索引） |
| 叶子节点链接 | 无 | 叶子节点通过链表相连 |
| 查找路径 | 不一定到达叶子 | 一定到达叶子节点 |
| 范围查询 | 需要中序遍历整棵树 | 直接遍历叶子链表 |
| 每节点容纳关键字 | 较少（因为存了数据） | 更多（不存数据） |
| 树的高度 | 较高 | 更矮 |

**B+树的优势：**
- 内部节点不存数据，同样大小的磁盘页可以容纳更多关键字，树更矮
- 叶子节点链表使得范围查询极其高效
- 查询性能更稳定（所有查询都走到叶子）
- 更适合磁盘存储（顺序 I/O 比随机 I/O 快得多）

## 常见错误

### 1. 将 B树与二叉树混淆

B树不是二叉树！B树是多路搜索树，每个节点可以有多个关键字和多个子节点。"B-Tree" 中的横杠不是 "B 减 Tree"，而是 "B-Tree" 这个整体名称。

### 2. 不理解磁盘块对齐

B树的节点大小通常与磁盘页大小对齐（4KB、16KB 等）。这是一个工程优化，而非算法要求。如果节点太小，浪费磁盘 I/O（一次读取只用了一小部分）；如果节点太大，一次读取可能跨页，需要两次 I/O。

### 3. 忘记 B+树叶子节点的链表结构

B+树的叶子节点通过双向链表相连，这是范围查询高效的关键。如果只关注树的结构而忽略叶子链表，就无法理解为什么 `SELECT * FROM t WHERE age BETWEEN 20 AND 30` 这样的查询能高效执行。

### 4. 混淆 B树的阶数定义

有些教材定义 m 阶 B树为每个节点最多 m-1 个关键字，有些定义为最多 m 个关键字。本文采用后者（每个节点最多 m 个子节点，m-1 个关键字），这是更常见的定义。

### 5. 误以为 B树只用于数据库

B树及其变体广泛用于文件系统、键值存储、全文索引等场景。任何需要高效磁盘查找的场景都可能用到 B树。

## 实际应用

### 1. 数据库索引

- **MySQL InnoDB**：使用 B+树作为聚簇索引和二级索引
- **PostgreSQL**：使用 B树作为默认索引类型（也支持其他类型）
- **Oracle**：默认索引类型就是 B树索引
- **SQLite**：使用 B+树存储表数据和索引

### 2. 文件系统

- **NTFS**：B+树管理文件索引（MFT）
- **ext4**：HTree（B树变体）管理目录
- **XFS**：B+树管理文件系统元数据
- **HFS+ / APFS**：B树管理文件系统

### 3. 键值存储

- **LMDB**（Lightning Memory-Mapped Database）：使用 B+树作为底层存储
- **RocksDB**：虽然主要使用 LSM 树，但也借鉴了 B树的思想
- **BerkeleyDB**：支持 B树作为存储引擎

### 4. 全文索引

- **Elasticsearch** 的倒排索引底层使用类似 B树的结构
- **Lucene** 使用 B树变体管理词典

## 总结

B树是一种为**磁盘存储优化**的多路平衡搜索树，核心特性：

- **多路分支**：每个节点存储大量关键字，大幅降低树的高度
- **自平衡**：所有叶子在同一层，查找性能稳定
- **磁盘友好**：节点大小与磁盘页对齐，最小化 I/O 次数
- **O(log_m n)** 的查找、插入、删除时间复杂度

B+树作为 B树最重要的变体，通过叶子链表和内部节点纯索引化，成为数据库索引的事实标准。

学习 B树的建议路径：先理解多路搜索树的概念，再掌握插入分裂机制，然后学习删除合并/重分配，最后对比 B树与 B+树的区别。理解 B树是理解现代数据库和文件系统工作原理的基础。
