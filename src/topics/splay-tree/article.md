# 伸展树 (Splay Tree)

## 概念解释

伸展树（Splay Tree）是一种自调整的二叉搜索树，由 Daniel Sleator 和 Robert Tarjan 于 1985 年提出。与 AVL 树、红黑树等需要维护额外平衡信息的数据结构不同，伸展树的每个节点只存储基本的值和左右子指针，不存储高度、颜色或任何平衡因子。

伸展树的核心思想是：**每次访问（查找、插入、删除）一个节点后，都通过一系列旋转操作将该节点移动到根部**。这个操作被称为"伸展"（Splay）。

伸展操作使用三种基本旋转模式：

- **Zig**：当目标节点的父节点就是根节点时，执行一次单旋（左旋或右旋），将目标节点提升为根。这是伸展操作的终止条件。
- **Zig-Zig**：当目标节点和父节点同为左子节点或同为右子节点时（即路径呈"直线"形状），先旋转父节点，再旋转目标节点。**注意：旋转顺序很重要，必须先旋转父节点。**
- **Zig-Zag**：当目标节点和父节点方向相反时（即路径呈"之字形"），先旋转目标节点，再旋转目标节点（等价于对父节点执行一次反向旋转后再对目标节点执行正向旋转）。

伸展树不保证树的高度严格平衡，但在连续操作下具有优秀的摊还性能。

## 为什么重要

伸展树在数据结构领域具有独特的地位，主要体现在以下几个方面：

### 1. 实现简单

与其他平衡二叉搜索树相比，伸展树的实现最为简洁。每个节点只需三个字段：值、左子指针、右子指针。不需要存储高度（AVL 树）、颜色（红黑树）或父指针。这使得伸展树的代码量通常只有 AVL 树或红黑树的一半。

### 2. 自适应访问模式

伸展树能够自动适应数据的访问模式。频繁被访问的节点会逐渐靠近根部，使得后续访问的代价更低。这种特性在访问模式具有局部性（Locality of Reference）时特别有效，例如：

- 某些数据被频繁访问（热点数据）
- 最近访问的数据很可能再次被访问（时间局部性）
- 与最近访问位置相近的数据很可能被访问（空间局部性）

### 3. 摊还 O(log n) 复杂度

虽然伸展树的单次操作可能需要 O(n) 时间（例如访问一条链状树的最深节点），但连续 m 次操作的总时间复杂度为 O(m log n)，即每次操作的摊还代价为 O(log n)。这一结论通过势能法（Potential Method）严格证明。

### 4. 理论价值

伸展树是"平衡猜想"（Balance Conjecture）的重要研究对象，推动了自调整数据结构理论的发展。它证明了不需要显式的平衡维护，仅通过局部调整就能达到优秀的全局性能。

## 核心原理

### 伸展操作的三种旋转

#### Zig 旋转（单旋）

当目标节点 x 的父节点 p 就是根节点时：

```
    p              x
   / \            / \
  x   C   →     A   p
 / \                / \
A   B              B   C
```

只需执行一次右旋（如果 x 是 p 的左子节点）或左旋（如果 x 是 p 的右子节点）。

#### Zig-Zig 旋转（同向双旋）

当 x 和 p 都是左子节点（或都是右子节点）时，**先旋转 p，再旋转 x**：

```
        g                x
       / \              / \
      p   D            A   p
     / \                  / \
    x   C       →       B   g
   / \                      / \
  A   B                    C   D
```

**关键：必须先旋转父节点 p（将 p 旋转到 g 的位置），再旋转 x（将 x 旋转到 p 的新位置）。** 这是伸展树中最容易出错的地方。

#### Zig-Zag 旋转（反向双旋）

当 x 是 p 的右子节点而 p 是 g 的左子节点（或相反）时：

```
      g                x
     / \              / \
    p   D       →   p   g
   / \              / \ / \
  A   x            A  B C  D
     / \
    B   C
```

等价于先对 p 执行左旋（将 x 提升到 p 的位置），再对 g 执行右旋（将 x 提升到 g 的位置）。

### 伸展操作的完整过程

以访问节点 2 为例：

```
初始树:        5
              / \
             3   7
            / \
           2   4
          /
         1

目标：将 2 伸展到根部

步骤 1: 2 的父节点 3 和祖父节点 5 都是左子节点 → Zig-Zig
        先旋转 3（父节点）:
            5              5
           / \            / \
          3   7   →      2   7
         / \            / \
        2   4          1   3
       /                    \
      1                      4

步骤 2: 2 现在是 5 的左子节点，父节点是根 → Zig
        旋转 2:
            5              2
           / \            / \
          2   7   →      1   5
         / \                / \
        1   3              3   7
             \              \
              4              4

最终: 2 成为根节点
```

### 摊还分析（势能法）

定义势能函数 Φ 为所有节点子树大小的对数之和：

```
Φ = Σ log(size(v))  对所有节点 v
```

其中 size(v) 是以 v 为根的子树中的节点数量。

关键引理（Access Lemma）：对包含 n 个节点的伸展树执行一次伸展操作，摊还代价为 O(log n)。

直觉解释：
- Zig-Zig 操作虽然需要两次旋转，但它显著降低了树的"不平衡度"（势能大幅下降）
- 势能的下降抵消了操作的实际代价
- 最终摊还代价被限制在 O(log n)

### 自调整特性

伸展树的"自调整"体现在：

1. **频繁访问的节点靠近根部**：多次访问同一个节点后，该节点会被反复伸展到根部
2. **访问序列的局部性被利用**：如果先访问节点 A，再访问 A 附近的节点 B，B 的伸展路径会因为 A 已经在根部附近而变短
3. **不需要预知访问模式**：与静态最优 BST 不同，伸展树在线地适应实际的访问序列

## 常见错误

### 1. Zig-Zig 旋转顺序错误

最常见的错误是在 Zig-Zig 情况下先旋转目标节点再旋转父节点。正确的顺序是**先旋转父节点，再旋转目标节点**。错误的旋转顺序会导致树的结构不正确，摊还复杂度的证明也不再成立。

```
正确: 先旋转 p，再旋转 x
错误: 先旋转 x，再旋转 p
```

### 2. 查找失败时不执行伸展

在某些实现中，开发者会在查找失败（目标值不存在）时直接返回，而不执行伸展操作。正确的做法是**将查找路径上最后访问的节点伸展到根部**。这样做可以保持伸展树的摊还性质，并且在某些应用场景下（如分裂操作）是必要的。

### 3. 与 AVL 树混淆

伸展树和 AVL 树虽然都是二叉搜索树，但设计理念完全不同：

| 特性 | 伸展树 | AVL 树 |
|------|--------|--------|
| 平衡信息 | 无 | 存储高度/平衡因子 |
| 平衡保证 | 摊还 O(log n) | 严格 O(log n) |
| 单次操作最坏 | O(n) | O(log n) |
| 旋转时机 | 每次访问后 | 插入/删除后 |
| 适用场景 | 访问模式有局部性 | 需要严格保证 |

### 4. 忘记处理空树或单节点情况

在实现伸展操作时，需要正确处理以下边界情况：
- 树为空时直接返回
- 目标节点已经是根节点时不需要伸展
- 目标节点是根的子节点时执行 Zig/Zag

### 5. 删除操作的实现错误

伸展树的删除操作需要：
1. 先将要删除的节点伸展到根部
2. 删除根节点后，将左子树中最大的节点伸展到左子树的根部
3. 将左子树的根连接为右子树的左子节点

如果省略第二步，合并左右子树后的树可能高度不平衡。

## 实际应用

### 1. 缓存系统

伸展树的自调整特性使其天然适合实现缓存。频繁访问的数据会自动"浮"到树的顶部，访问代价更低。这与 LRU（最近最少使用）缓存的思想类似，但伸展树提供了更灵活的键值操作。

### 2. 内存分配器

在内存分配器中，伸展树可以用来管理空闲内存块。频繁分配和释放的内存块会自动靠近根部，加速分配操作。Doug Lea 的 malloc 实现中就使用了类似的思想。

### 3. Link-Cut Tree（竞赛编程）

伸展树是实现 Link-Cut Tree 的核心组件。Link-Cut Tree 是一种用于动态树问题的数据结构，支持：
- 链接（Link）：连接两棵树
- 切割（Cut）：断开一条边
- 路径查询：查询两点之间路径上的信息

在竞赛编程中，Link-Cut Tree 常用于解决动态树、最小生成树的动态维护等问题。

### 4. 区间操作

通过伸展树的分裂（Split）和合并（Merge）操作，可以高效地实现区间翻转、区间求和等操作。这在某些竞赛题目中非常有用。

### 5. 网络路由表

在某些网络路由协议中，伸展树可以用于维护路由表。由于网络流量往往具有局部性（某些路由条目被频繁查询），伸展树的自调整特性可以提高路由查找的效率。

## 总结

伸展树是一种优雅的自调整二叉搜索树，其核心思想是"每次访问后将节点伸展到根部"。它不需要存储任何额外的平衡信息，实现简洁，但通过摊还分析保证了 O(log n) 的操作效率。

伸展树的最大优势在于其自适应性：频繁访问的节点会自动靠近根部，使得在访问模式具有局部性时表现优异。虽然单次操作可能达到 O(n) 的最坏情况，但在实际应用中，这种最坏情况很少出现，且摊还性能保证了整体效率。

伸展树特别适合以下场景：
- 访问模式不均匀，存在热点数据
- 需要简洁的实现
- 可以接受摊还复杂度保证（而非严格最坏情况保证）
- 竞赛编程中的 Link-Cut Tree 等高级数据结构

与其他平衡 BST 相比，伸展树的取舍是：用最坏情况的严格保证换取实现的简洁性和对访问模式的自适应能力。理解伸展树有助于深入认识"自调整"这一数据结构设计范式，以及摊还分析这一重要的算法分析方法。
